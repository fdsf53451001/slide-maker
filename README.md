# Slide Maker

**Local-first, image-based presentation generation.** Slide Maker turns a brief, your own
source documents and a reusable visual style into a full slide deck ‚Äî one image per page,
generated by whichever image model you point it at ‚Äî and gives you a page-oriented editor to
review, regenerate, retouch and export the result.

Everything runs on your machine: an Express API plus a React editor served from the same
port, with projects, styles, versions and jobs persisted as plain files under `.data/`.

## Why image-based slides

Conventional deck generators emit shapes and text boxes, which caps the visual quality at
whatever the template can express. Slide Maker instead treats each page as a **single
1920√ó1080 image** produced by an image model against a provider-neutral contract, so the page
can look like anything a designer would draw ‚Äî and then layers the things a model must not be
trusted to draw (page numbers, editable text) on top as system-composited chrome.

The trade-off is deliberate: pages are pixels, so editing is done by regenerating, masked
editing, or the extracted text layer ‚Äî not by dragging text boxes.

## Features

**Authoring workflow**

- Two-step persisted flow: **brief ‚Üí outline**, then **settings ‚Üí generate deck**. The
  confirmed outline determines the page count; confirming step two queues every page at once.
- Model-written outlines with optional live web search, where every citation must come from a
  page whose body text was actually fetched ‚Äî model-written search summaries never count as a
  verified source.
- Per-page editing of title, bullets, speaker notes, layout intent and per-page source
  selection.
- Page add, duplicate, delete and drag-to-reorder.

**Sources**

- Upload PDF, PPTX, DOCX, Markdown, plain text, PNG and JPEG. Text is extracted
  project-locally into stable chunks (PDF extraction restores table layout as Markdown pipe
  tables where it can detect one).
- Full-text search over chunks via SQLite FTS5 (trigram tokenizer), rebuilt from the project
  on every boot.
- Web sources are captured with a shared URL-safety policy before they can be cited.

**Styles**

- Server-side style library with immutable versions, duplication and restore.
- Reference images, including "pull the first pages of a PDF as style thumbnails".
- Style analysis that turns an uploaded reference into a style preset.
- Each project carries a **style snapshot**, so editing a style later never mutates decks
  already generated.

**Generation**

- Persistent per-slide jobs with bounded provider concurrency, cancellation, crash recovery
  and batch generation. Jobs that die with the server are reclassified (`SERVER_SHUTDOWN` /
  `SERVER_RESTARTED`) on the next boot instead of hanging forever.
- Immutable image history per page: every generation, edit and restore is a new version you
  can switch back to; versions can be deleted with their assets reclaimed.
- Masked editing ("change just this region") and instruction-based edits, where the transport
  supports them.

**Pixels back to text**

- **Extract text**: PaddleOCR finds the text on a generated page, the text is lifted into an
  editable DOM layer, and the background is inpainted ‚Äî by default with a local OpenCV
  inpainter (fast, zero quota), or optionally by the image model through a masked edit.
- **PDF deck import**: bring an existing 16:9 PDF in as a project, model-free. Text comes from
  the PDF's own text layer and the clean background comes from re-rendering the page with text
  operators filtered out. Two versions are created per page ‚Äî the untouched original and the
  editable-text variant ‚Äî so you can always fall back to pixel-perfect originals.

**Deck chrome and export**

- Project-level page numbers composited by the system, not drawn by the model: position,
  format (`7`, `7 / 20`, `Á¨¨ 7 ÈÝÅ`), start number, skip-cover, size, colour, opacity and an
  optional background chip. The same geometry module drives the canvas overlay, the exported image
  and the PPTX text box, so all three agree.
- Export to **PPTX**, **PDF**, an ordered **PNG ZIP**, or a portable **`.slide-project`**
  archive that imports back into another instance. Lossless PNG is preserved where fidelity
  matters; PPTX/PDF embed quality-88 4:4:4 JPEG to keep decks a sane size.
- Presentation mode in the editor.

**Providers**

- A deterministic **mock** image provider is the default: no network, no quota, useful for
  development and tests.
- Production transports for OpenAI-compatible endpoints (CLIProxyAPI, OpenAI, LiteLLM,
  OpenRouter‚Ä¶), native Gemini (AI Studio), and an experimental Codex app-server spike.
- A **model library** in the UI: connections (base URL + key + protocol), per-capability model
  entries (image / text / search), and named combinations you bind to a project. API keys are
  redacted on read.
- The cheapest practical setup is a local CLIProxyAPI gateway with `gpt-image-2` ‚Äî see
  [Connecting a model provider](#connecting-a-model-provider).

## Quick start

```bash
npx pnpm@10.13.1 install
npx pnpm@10.13.1 dev
```

Open <http://localhost:4173>. The local server serves the React editor and the API together.
`pnpm dev` is long-running and stays attached to the terminal until you stop it (`Ctrl+C`);
its startup messages report which providers are active and never print credentials.

For UI-only work, run `pnpm dev:web` in a second terminal and open Vite's URL instead.

Project data is written to `.data/` by default (override with `SLIDE_MAKER_DATA_ROOT`) and is
intentionally ignored by Git.

### Optional: OCR and local inpainting

Needed only for **Extract text** on generated pages. PDF deck import does not use it.

```bash
pnpm setup:ocr     # creates .venv-ocr with paddlepaddle + paddleocr, downloads model weights
```

The script picks the newest available `python3.12`‚Ä¶`python3.9`. Restart the server afterwards;
`GET /api/ocr/status` reports availability.

## Connecting a model provider

There are four ways to give Slide Maker an image model. They differ in cost, setup effort and
which capabilities you get.

| #   | Way to connect                                                                              | Setup                                                 | Best for                                        |
| --- | ------------------------------------------------------------------------------------------- | ----------------------------------------------------- | ----------------------------------------------- |
| 1   | **Mock provider** (default)                                                                 | none                                                  | Development, tests, trying the workflow offline |
| 2   | **OpenAI-compatible gateway** ‚Äî CLIProxyAPI (CLI2Proxy), LiteLLM, OpenRouter, OpenAI itself | Run/point at a gateway, then one connection in the UI | **Recommended for real decks**                  |
| 3   | **Gemini AI Studio, native**                                                                | An AI Studio API key                                  | Gemini image models without a gateway           |
| 4   | **Codex CLI app-server**                                                                    | A locally authenticated Codex CLI, opt-in flag        | Experimental; see the security notes            |

### Recommended: CLIProxyAPI + `gpt-image-2`

The cheapest setup that still produces good decks is a local **CLIProxyAPI (CLI2Proxy)**
gateway with **`gpt-image-2`** on the Images API transport. It reuses your existing CLI
subscription instead of billing per-image API calls, and `gpt-image-2` on
`/images/generations` + `/images/edits` covers the whole feature surface ‚Äî generation,
reference images (as an `image[]` array on the edit endpoint), and masked edits.

In the model library, create:

- **Connection** ‚Äî protocol `openai`, base URL `http://localhost:8317/v1` (whatever your
  gateway listens on), API key as configured.
- **Image model** ‚Äî kind `openai`, model `gpt-image-2`, image API `images`.
- **Text / search models** ‚Äî point them at the same connection, or leave them on Codex.
- **Combination** ‚Äî bundle the three and set it as the default.

Equivalent first-boot environment seed, if you prefer to start from a config:

```bash
SLIDE_MAKER_OPENAI_BASE_URL=http://localhost:8317/v1 \
SLIDE_MAKER_OPENAI_API_KEY=... \
SLIDE_MAKER_OPENAI_IMAGE_MODEL=gpt-image-2 \
SLIDE_MAKER_OPENAI_IMAGE_API=images \
SLIDE_MAKER_OPENAI_TEXT_MODEL=... \
pnpm dev
```

Pick a different transport when you have a reason to: `chat` for GPT tool-image or native
`gemini-3.1-flash-image` (up to 8 ordered reference images), `openrouter-image` for
OpenRouter's `/images` shape, or the native Gemini provider when you want Gemini's image
models and Google Search grounding.

## Configuring models

The UI is the primary path. Open the **model library** in the editor and add:

1. a **connection** ‚Äî base URL, API key, and protocol (`openai` for anything
   OpenAI-compatible, `gemini` for AI Studio's native endpoint);
2. **models** ‚Äî one entry per capability (`image`, `text`, `search`), each referencing a
   connection;
3. a **combination** ‚Äî one image + text + search model, bound to a project or set as default.

Settings live in `SLIDE_MAKER_DATA_ROOT/models.json`, which is the single source of truth once
it exists. Environment variables only seed that file on first boot when it is absent, so
changing them later has no effect ‚Äî edit the library in the UI instead.

> `models.json` stores API keys in plaintext. Protect the data directory accordingly.

### Image transports

All transports share the same provider-neutral **Codex-baseline image contract** in
`@slide-maker/core`: canvas, complete slide fields, style snapshot, information density,
edit/mask semantics, ordered reference roles, direct-asset fidelity, the untrusted-data
boundary, and an explicit ban on the model drawing deck chrome. Adapters add only their
invocation and response-format rules; every accepted raster is normalized to a canvas-sized
PNG.

| Transport                   | Module                              | Endpoint                               | Example model                             | Reference images                          |
| --------------------------- | ----------------------------------- | -------------------------------------- | ----------------------------------------- | ----------------------------------------- |
| OpenAI Images API (default) | `provider-openai/src/image-api.ts`  | `/images/generations`, `/images/edits` | `gpt-image-2`                             | Via `/images/edits`; mask edits supported |
| OpenAI-compatible Chat      | `provider-openai/src/image-chat.ts` | `/chat/completions`                    | `gpt-5.6-terra`, `gemini-3.1-flash-image` | Yes, ordered, up to 8                     |
| OpenRouter images           | `provider-openai`                   | `/images`                              | provider-specific                         | Yes, via `input_references`               |
| Gemini native               | `@slide-maker/provider-gemini`      | `:generateContent`                     | `gemini-2.5-flash-image`                  | Yes, ordered inline data                  |
| Codex app-server            | `@slide-maker/provider-codex`       | Codex CLI app-server                   | configured Codex image model              | Yes                                       |
| Mock                        | `@slide-maker/provider-mock`        | none                                   | ‚Äî                                         | ‚Äî                                         |

Notes from live testing:

- Google's **OpenAI-compatibility layer cannot be used for images**: it fails to serialize
  image output, has no `/images/edits`, and rejects every Google Search grounding syntax. Use
  the native Gemini provider.
- Gemini has no separate edit/mask endpoint ‚Äî a mask is simply one more input image, with the
  semantics carried entirely by the image contract.
- `gemini-3-flash-agent` is not a usable image-output model for this flow.

### Text and web search

Text generation and web search are decoupled: a `WebSearchProvider` fetches and verifies
sources first, then a pure-reasoning `StructuredTextProvider` consumes them. Non-strict
gateways (Gemini-family in particular) ignore `json_schema`, so those providers add a
JSON-only system instruction and parse leniently.

### Environment variables

Server variables are all defined in `apps/server/src/config.ts` and validated at startup ‚Äî
an illegal value throws rather than silently degrading.

| Variable                                                                                                               | Purpose                                                                                                         |
| ---------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| `PORT`, `HOST`                                                                                                         | Listen address (default `127.0.0.1:4173`)                                                                       |
| `SLIDE_MAKER_DATA_ROOT`                                                                                                | Project data directory (default `.data/`)                                                                       |
| `SLIDE_MAKER_SEARCH_INDEX_PATH`                                                                                        | Move the SQLite FTS index off the data root                                                                     |
| `SLIDE_MAKER_TRUSTED_HOSTS`                                                                                            | Extra allowed hostnames; unset, only localhost is served                                                        |
| `SLIDE_MAKER_ENABLE_CODEX_SOFT_SANDBOX`                                                                                | `1` enables the Codex image provider (see below)                                                                |
| `SLIDE_MAKER_CODEX_TIMEOUT_MS`                                                                                         | 30 000 ‚Äì 1 800 000, default 600 000                                                                             |
| `SLIDE_MAKER_CODEX_MAX_CONCURRENCY`                                                                                    | 1 ‚Äì 4, default 3                                                                                                |
| `SLIDE_MAKER_CODEX_MODEL`, `SLIDE_MAKER_CODEX_REASONING_EFFORT`                                                        | Override the Codex CLI's own settings                                                                           |
| `SLIDE_MAKER_OPENAI_BASE_URL`, `_API_KEY`, `_IMAGE_MODEL`, `_TEXT_MODEL`, `_SEARCH_MODEL`, `_IMAGE_API`, `_TIMEOUT_MS` | First-boot seed for an OpenAI-compatible endpoint                                                               |
| `SLIDE_MAKER_TEXT_ENGINE`, `SLIDE_MAKER_WEB_SEARCH_ENGINE`                                                             | `codex` (default) or `openai`                                                                                   |
| `SLIDE_MAKER_OCR_MODEL_TIER`, `_OCR_DET_SIDE_LEN`, `_OCR_PYTHON`, `_OCR_SCRIPT`                                        | PaddleOCR tier (`tiny`/`small`/`medium`, default `medium`), detection side length, interpreter and script paths |
| `SLIDE_MAKER_INPAINT_PYTHON`, `_INPAINT_SCRIPT`                                                                        | Local OpenCV inpainter paths                                                                                    |
| `SLIDE_MAKER_LOG_EGRESS_IP`                                                                                            | `1` logs the outbound IP at startup                                                                             |

## Development

```bash
pnpm install          # npx pnpm@10.13.1 install if you don't have pnpm
pnpm dev              # build every package, then run the server at :4173
pnpm dev:web          # Vite dev server for UI-only work
pnpm check            # typecheck + test + build ‚Äî the full gate
pnpm test             # vitest across all packages
pnpm format           # Prettier
```

Run a single package's tests with `pnpm --filter @slide-maker/server test`, or a single file
with `pnpm --filter @slide-maker/server exec vitest run test/config.test.ts`.

Two things worth knowing before you change anything:

- Dev mode relies on `NODE_OPTIONS=--conditions=development` together with
  `customConditions: ["development"]` in tsconfig, so workspace packages resolve to `src/*.ts`.
  Without it you silently get stale `dist/` output.
- TypeScript runs with `noUncheckedIndexedAccess` and `exactOptionalPropertyTypes`: indexed
  access yields `undefined`, and optional properties may not be explicitly set to `undefined`.

The `smoke:*` scripts (`smoke:image:codex`, `smoke:deck:grok`, ‚Ä¶) are **live** end-to-end
tests that consume real model quota. They are deliberately excluded from `pnpm check`.

## Project structure

pnpm monorepo; internal dependencies use `workspace:*`.

| Package                    | Role                                                                                               |
| -------------------------- | -------------------------------------------------------------------------------------------------- |
| `apps/server`              | Express 5 API, job runner, exporters, OCR/inpaint bridges, PDF pipelines                           |
| `apps/editor`              | React 19 + Vite editor. Double build: standalone app and embeddable library                        |
| `packages/core`            | Zod schemas, project helpers, provider contracts, image contract, page-number geometry, URL safety |
| `packages/provider-mock`   | Deterministic, no-cost image provider (the default)                                                |
| `packages/provider-openai` | OpenAI-compatible image / structured-text / web-search adapters                                    |
| `packages/provider-gemini` | Gemini AI Studio native `:generateContent` adapters                                                |
| `packages/provider-codex`  | Experimental Codex app-server spike, pinned to protocol `0.144.4`                                  |

`packages/core` is the single source of truth for anything three surfaces must agree on ‚Äî
image semantics, page-number geometry, URL safety. Do not re-implement those in an adapter.

### Embedding the editor

`@slide-maker/editor` also builds as a library:

```tsx
import { Editor, ModelLibrary } from "@slide-maker/editor";
import "@slide-maker/editor/styles.css";

export default function App() {
  return <Editor />;
}
```

The editor calls the API with same-origin relative paths, so it must be served from the same
origin as `apps/server`. React 19 is a peer dependency.

## Deployment

`infra/` contains Terraform for Cloud Run with a gcsfuse-mounted Cloud Storage bucket, fronted
by IAP, sized for a handful of trusted users sharing one data set. Several of its settings ‚Äî
single instance, always-allocated CPU, hierarchical-namespace bucket ‚Äî are correctness
requirements rather than tuning knobs.

See **[docs/DEPLOYMENT.md](docs/DEPLOYMENT.md)** for the full procedure and its caveats.

## Security notes

- The API refuses any request whose host is not localhost (`LOCAL_HOST_REQUIRED`) unless you
  explicitly list hostnames in `SLIDE_MAKER_TRUSTED_HOSTS`. There is no built-in
  authentication ‚Äî put a proxy in front of it if you expose it.
- API keys are stored in plaintext in `models.json` under the data root.
- The **Codex image provider is opt-in** (`SLIDE_MAKER_ENABLE_CODEX_SOFT_SANDBOX=1`) and
  offers **soft isolation, not a security boundary**. It pins the experimental Codex `0.144.4`
  app-server protocol and requests a read-only filesystem policy, no approvals, no turn
  network access and an ephemeral thread, rejecting file changes, MCP calls, non-`exec`
  dynamic tools, web search, unexpected response policies and uncorrelated events. Even so,
  app-server still loads the real `CODEX_HOME` configuration and tool surfaces, so a malicious
  reference or prompt can still cause prompt injection, local-data disclosure, configured-tool
  side effects or quota consumption. Run it only in a disposable account or container with no
  secrets and no privileged tools.
- Model output is treated as untrusted: Codex results must be a regular, non-symlink PNG
  inside the job workspace and are checked for size, dimensions, chunk bounds, the required
  IHDR/IDAT/IEND chunks and CRCs before being re-rendered to exactly 1920√ó1080 and validated
  again. Source documents and web pages are marked as untrusted data in prompts.
